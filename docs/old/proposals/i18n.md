# i18n as a first-class citizen

> Status: **proposal**. No DSL surface or runtime yet exists. This doc
> defines the layered design, library picks, key/lock model, and the
> three-way-merge regeneration flow. Implementation phasing at the end.
>
> The companion note [`i18n-strings.md`](./i18n-strings.md) defines the
> string-composition surface (template literals, ICU pass-through, the
> ban on `+` in user-visible slots, and placeholder-rename handling).
> Read that one *with* this one — together they form a single design.

## Problem

A Loom system has user-facing text in three places, none of which today
acknowledge that translation is a concern:

1. **DSL source** — `Heading { "Action showcase" }`,
   `menu { label: "Sales" }`, `precondition: …`. End users read these.
   ([web/src/examples/action-showcase.ddd:58](../../../web/src/examples/action-showcase.ddd),
   [examples/sales-ui.ddd:30](../../../examples/sales-ui.ddd))
2. **Generated chrome** — `"Cancel"`, `"Save"`, `"{op} succeeded"`,
   `"This operation has no parameters."` baked into design-pack
   templates.
   ([designs/mantine/v7/form-op-module.hbs:5](../../../designs/mantine/v7/form-op-module.hbs),
   [designs/ashPhoenix/v3/primitive-form-of.heex.hbs:5](../../../designs/ashPhoenix/v3/primitive-form-of.heex.hbs))
3. **Diagnostics** — validator messages
   ([src/language/validators/structural.ts:208](../../../src/language/validators/structural.ts))
   and generated-domain validation messages
   ([src/generator/typescript/render-stmt.ts:81](../../../src/generator/typescript/render-stmt.ts),
   [src/generator/typescript/zod-refine.ts:84](../../../src/generator/typescript/zod-refine.ts)).

A repo-wide search for `i18n`, `intl`, `react-intl`, `i18next`,
`translate`, `gettext` finds nothing. Zero infrastructure today. This
proposal makes translation a property of the compiler and the generated
code, not a per-app retrofit.

## Goals

- **Zero-effort default.** A fresh `ddd generate system` produces a
  working English app with a complete catalog. No developer step
  required.
- **No translatable strings in code, ever.** All translations live in
  JSON catalogs. The DSL only ever holds source-language strings; the
  generated code only ever holds keys.
- **One catalog format, every backend.** ICU MessageFormat per-locale
  JSON. React/Hono/.NET/Phoenix differ only in their runtime adapter.
- **Server↔client lockstep on validation messages.** A Zod
  refinement, a FluentValidation rule, an Ecto changeset error, and a
  React form error all resolve through the same catalog by the same key.
- **Git-native translator workflow.** Translators clone the repo, edit
  one file per locale, PR it. No SaaS, no plug-ins, no out-of-band sync.
- **Opt-in, deliberate regeneration.** Re-running `ddd generate` does
  not silently rewrite the translator's work. Reconciliation is an
  explicit checkpoint (`ddd i18n sync`) backed by a three-way merge
  against a committed lock file.

## Non-goals (this proposal)

- Localised compiler diagnostics. Loom's authors are developers; English
  error messages are acceptable for v1. Centralisation into a keyed
  message catalog is a deferred prerequisite, not part of this work.
- RTL layout, date/number/currency *format* selection beyond what ICU
  already provides at the runtime layer.
- Translation memory / machine-translation integration. Out-of-band; a
  translator can run any tool against the locale files.
- Per-tenant locale overrides. Single static catalog per build.

## Layered design

Three layers, three corresponding decisions.

### Layer 1 — DSL source text (the 90% case)

Two complementary surfaces:

**Implicit (default).** Walker-stdlib slots that hold user-visible text
(`Heading.text`, `Text.text`, `Button.label`, `Empty.text`, `Card.title`,
`Badge.text`, page `title:`, menu `label:` / `section:`) are tagged
user-visible in [src/generator/_walker/registry.ts](../../../src/generator/_walker/registry.ts)
and mirrored in [src/language/walker-stdlib.ts](../../../src/language/walker-stdlib.ts).
At lowering time (phase ⑤b,
[src/ir/lower-expr.ts](../../../src/ir/lower-expr.ts)), every `StringLit`
appearing in such a slot gets `StringLitIR.userVisible = true`. No
syntax change.

```ddd
page OrderShow(o: Order) {
  Heading { "Order summary" }      // extracted as user-visible
  Empty   { "No items yet" }       // extracted as user-visible
}
```

**Explicit `text { … }` block.** For strings reused across pages, or
strings whose stability matters more than the literal value (legal
notices, marketing copy, anything that survives a UI refactor), authors
declare named text:

```ddd
context Sales {
  text {
    orderNotFound: "Order not found"
    cancelConfirm: "Are you sure you want to cancel {count, plural, one {# order} other {# orders}}?"
  }

  page OrderShow(o: Order) {
    Empty { orderNotFound }
  }
}
```

Named entries become `text.<context>.<name>` in the catalog — stable
across rename of the page, reorder of its children, and rephrasing of
adjacent strings. Inline literals are keyed by a content hash (below).

**Why both.** Inline is friction-free for the 90% case where the string
*is* the canonical thing. Named is for the 10% where the *reference* is
the canonical thing and the string is implementation. Authors choose
per-string. This mirrors how Loom already lets authors pick between
inline `view` filters and named `view` declarations.

**String composition is template-literal-only.** Inside any user-visible
slot, `+` is rejected; values are spliced via `${…}` interpolation that
lowers to ICU placeholders. Format codes, plurals, and `select` cases
pass through to ICU verbatim. The full design — failure modes of `+`,
placeholder-name derivation, key stability under field renames, the
worked before/after — is in [`i18n-strings.md`](./i18n-strings.md). This
proposal assumes that surface throughout.

### Layer 2 — generated chrome

Each design pack ships an ICU JSON catalog of its own for the strings
the pack's `.hbs` snippets need: form action labels, success/error
toasts, empty-state placeholders, navigation chrome:

```
designs/mantine/v7/
  i18n/
    en.json
    pt-BR.json
    ja.json
  form-op-module.hbs    ← references keys, not literals
```

Pack catalogs merge into the system catalog at codegen time
(phase ⑨), namespaced under `chrome.<pack>.<key>` so they cannot
collide with model-extracted keys. A pack contributing only English is
fine — the runtime falls back to source.

### Layer 3 — diagnostics

Validator messages stay inline English for v1. The change is purely
mechanical preparation: move the message *strings* into
`src/language/validators/messages.ts` keyed by their existing diagnostic
codes (e.g., `loom.bare-aggregate-in-type`,
[src/language/ddd-scope.ts](../../../src/language/ddd-scope.ts)). Future
localisation becomes a JSON overlay; no validator code change required.
This proposal commits to the move but not to translation.

## Library picks

### Wire format on disk — ICU MessageFormat, JSON

ICU is the lingua franca translators expect. Placeholders, plurals,
gender, ordinals, selectors, formatted dates/numbers/currencies all in
one syntax. Storing as JSON keeps diffs surgical and tooling universal.

```json
{
  "page.OrderList.heading.a3f2": "Orders",
  "invariant.Order.totalPositive":
    "Order total must be at least {minTotal, number, ::currency/USD}",
  "chrome.mantine.form.cancel": "Cancel",
  "chrome.mantine.form.opSucceeded": "{op} succeeded"
}
```

### React runtime — react-intl (FormatJS)

| | react-intl | i18next + plugin |
|---|---|---|
| ICU support | Native | Plugin (`i18next-icu`) |
| Extraction | `@formatjs/cli extract` produces typed JSON | Manual or `i18next-parser` |
| Bundle (gzip) | ~25 kB | ~15 kB core + ~5 kB ICU plugin |
| TS story | Strong — generated types per locale | Stringly-typed by default |
| React API | `<FormattedMessage id="..." values={...} />` | `t('key', vars)` |

react-intl wins for Loom because **Loom emits the React code** —
developer ergonomics of `t()` doesn't matter, ICU correctness and a
deterministic extraction pipeline do. FormatJS's extraction format is
the same shape as our `wire-spec.json` artefact
([src/system/wire-spec.ts](../../../src/system/wire-spec.ts)) and slots
into the same workflow.

### Non-React backends — same JSON, different adapter

- **Hono** → `intl-messageformat` (FormatJS's core, no React) in a
  small `src/i18n.ts` shim.
- **.NET** → `IStringLocalizer` with a custom
  `IStringLocalizerFactory` that reads the same JSON.
- **Phoenix** → Gettext stays the runtime; codegen translates
  ICU JSON → `.po` at emit time. (Alternative: `:cldr_messages` for
  direct ICU runtime in Elixir. Decide per pack.)

One catalog format on disk, four runtime adapters. Translators learn
one format.

### Zod / runtime validation — message-as-key

Today [src/generator/typescript/zod-refine.ts:84](../../../src/generator/typescript/zod-refine.ts)
emits `message: \`Invariant violated: ${inv.source}\`` — raw English
that leaks DSL source to the client. Replace with stable keys:

```ts
z.object({
  total: z.number().refine(v => v > 0, {
    message: 'invariant.Order.totalPositive',
    path: ['total'],
  }),
})
```

The React form's `errorMap` (or a thin wrapper around `safeParse`) reads
`error.message` as a catalog key and renders via
`<FormattedMessage id={error.message} values={error.params} />`. Server
`DomainError` carries the same `{ code, params }` shape; the React
boundary translates either source.

Zod's *own* errors (`required`, `invalid_type`, `too_small`) ship via
`zod-i18n-map` — a ~5 kB community library with maintained catalogs for
~30 locales. Loom emits one `import` in the generated app shell.

Invariant/precondition messages become regular user-visible strings:

```ddd
aggregate Order {
  invariant totalPositive {
    total > 0
    message: "Order total must be at least {minTotal, number, ::currency/USD}"
  }
}
```

The `message:` slot's `StringLitIR.userVisible = true` flows through
the same extraction pipeline as `Heading` text — no special-casing. If
omitted, Loom synthesises a key (`invariant.<Aggregate>.<name>`) and
falls back to `humanize(name)`; works without translator involvement,
translatable when someone bothers.

## Keys — stability is the whole game

Three categories, each with a different key-encoding strategy:

| Source | Key shape | Stability |
|---|---|---|
| Auto-humanised field labels | `<aggregate>.<field>` (`Order.customerName`) | Stable until field rename |
| Inline `Heading { "..." }` etc. | `page.<page>.<role>.<sha-6>` | Stable until the *string* changes |
| Explicit `text { name: "..." }` | `text.<context>.<name>` | Stable until *name* changes |

**Content-hashing inline strings is the key (pun intended) trick.** A
literal's key embeds a 6-char base64 of `sha512(source)`. Effects:

- Reordering siblings → keys unchanged. (Position is not in the key.)
- Renaming the page → keys change. (Page name *is* in the key; rare and
  intentional.)
- Editing the string → key changes. The old key disappears, a new key
  appears. Cleanly handled as delete-old + add-new in a sync diff —
  *no merge conflict on the value*. Translator sees one go, one arrive,
  carries the work across.

This matches FormatJS's default extractor behaviour
(`idInterpolationPattern: '[sha512:contenthash:base64:6]'`) — we are
not inventing anything.

Named `text` entries get true stable keys precisely so that authors who
*want* the translation to survive a string rephrase have a way to
express that: name it, keep the name, edit the value.

## Files — what lives where

```
<repo>/
  examples/sales-ui.ddd                 ← source of truth for default strings
  locales/                              ← translator-owned tree
    .loom/
      source.lock.json                  ← machine, sync-time snapshot; BASE for merges
    fr.json                             ← human, sync-merged
    ja.json                             ← human, sync-merged
  out/                                  ← regenerated; gitignored except .loom/
    .loom/
      messages.en.json                  ← machine, every-build snapshot; THEIRS
      wire-spec.json                    ← existing, unchanged
    apps/orders/src/locales/
      en.json                           ← copy of out/.loom/messages.en.json
      fr.json                           ← copy of <repo>/locales/fr.json
      ja.json                           ← copy of <repo>/locales/ja.json
```

Three machine artefacts, two human artefacts, distinct write surfaces:

| File | Written by | When |
|---|---|---|
| `out/.loom/messages.en.json` | `ddd generate` | Every build |
| `out/apps/*/src/locales/*.json` | `ddd generate` | Every build (pure copies) |
| `<repo>/locales/.loom/source.lock.json` | `ddd i18n sync` / `init` | Only on opt-in |
| `<repo>/locales/<locale>.json` | `ddd i18n sync` / `init` | Only on opt-in |

**Codegen never writes under `<repo>/locales/`.** Reconciliation is a
deliberate checkpoint, not a side effect.

## Regeneration — opt-in, three-way merge

The lock is the BASE; the current extraction is THEIRS; the human file
is OURS:

```
BASE    locales/.loom/source.lock.json   ← snapshot of source at last sync
OURS    locales/<locale>.json            ← translator's current file
THEIRS  out/.loom/messages.en.json       ← what codegen extracted now
        ↓ ddd i18n sync (per-key 3-way merge)
        new locales/<locale>.json (with conflict markers if any)
        new locales/.loom/source.lock.json  ← snapshot updated to THEIRS
```

### The four cases

| Change | BASE | THEIRS | OURS (fr) | Merge result |
|---|---|---|---|---|
| **New key** | — | `"Shipments"` | — | Add `"TODO: Shipments"` to fr |
| **Deleted key** | `"Orders"` | — | `"Commandes"` | Drop from fr (or `_stale` — flag, configurable) |
| **Unchanged + user translated** | `"Orders"` | `"Orders"` | `"Commandes"` | Keep `"Commandes"`; no change |
| **Source changed + user translated** | `"Orders"` | `"Order management"` | `"Commandes"` | See below |

The fourth case is the only hard one, and content-hashed keys convert
it from a conflict into a clean delete-old + add-new for inline
strings:

```diff
- "page.OrderList.heading.a3f2": "Commandes"      (was "Orders")
+ "page.OrderList.heading.b7c1": "TODO: Order management"
```

`sync` offers to carry the old translation forward as a starting point:

```
$ ddd i18n sync
locales/fr.json:
  Source changed for "Orders" → "Order management"
  Carry old translation forward as starting point? [Y/n]
  → writes "page.OrderList.heading.b7c1": "Commandes (was: Orders)"
```

For named `text` entries (stable key) where the source *did* change,
git-style conflict markers in the JSON are the answer:

```json
{
  "text.Sales.orderNotFound":
<<<<<<< OURS
    "Commande introuvable"
||||||| BASE
    "Order not found"
=======
    "We couldn't find that order"
>>>>>>> THEIRS,
```

JSON won't parse until resolved → CI catches it → translator opens
their editor, picks one side or rewrites, commits. Standard merge
muscle memory.

### Why opt-in, not automatic

Three reasons:

1. **Codegen must stay idempotent on the source tree.** Re-running
   `ddd generate` should be safe to do a thousand times in CI without
   ever touching what humans wrote.
2. **Reconciliation is a commit-worthy intent**, like adding a
   migration. The PR diff *is* the record.
3. **Conflicts need a human.** A silent merge in a build step would
   resolve them invisibly; a deliberate `sync` surfaces them in the
   diff.

### Why the BASE must lag

Three-way merge only produces information if BASE is genuinely older
than THEIRS. If we rewrote the lock on every codegen, BASE would always
equal THEIRS and the merge would have nothing to act on. The lock is
updated *only* when `sync` successfully completes — at that moment, the
translator has reconciled with the new source, so the new source
becomes the new BASE for next time.

## CLI surface

```bash
ddd i18n init <locale>          # scaffold locales/<locale>.json + lock if absent
ddd i18n sync                   # three-way merge for every locale; update lock
ddd i18n sync --locale fr       # one locale
ddd i18n status                 # what sync would do; exit non-zero if anything pending
ddd i18n check --strict         # CI gate: fail on TODO:, conflict markers, missing keys
ddd i18n prune                  # delete keys absent from source (off by default)
ddd i18n extract                # write out/.loom/messages.en.json without full codegen
                                # (useful in CI to validate without running every backend)
```

`sync` is the one command translators and developers learn. It is
`git merge` for strings. The lock is the only piece of vocabulary they
need, and only when something conflicts.

## Pre-commit + CI

**Pre-commit hook** (optional, recommended) auto-runs `ddd i18n sync`
and stages the result. Developer who writes
`Heading { "Shipments" }` and commits: locale files get `TODO:` entries
in the same commit, automatically. They never forget — the diff is
right there in the PR.

**CI** runs `ddd i18n check --strict --locale fr` per locale tagged
production-blocking. Configurable per locale: `fr` strict, `ja` lenient
during build-out, pseudo-locales (`xx-pseudo`) always strict. Pseudo
locale (`[!!Action showcase!!]` style) generated on demand to spot
un-extracted strings in QA.

## Pipeline integration

Folds into the existing ten-phase pipeline ([CLAUDE.md](../../../CLAUDE.md))
without crossing layer boundaries:

| Phase | Change |
|---|---|
| ④ AST validate | New diagnostic `loom.user-visible-concat` (see below). |
| ⑤b lower-expr | Add `StringLitIR.userVisible?: true`; set in user-visible slots. Stamp invariant/precondition `message:` strings the same way. |
| ⑥ enrich | New `enrichI18nCatalog`: walk user-visible strings, derive `<key, source>` pairs, attach as `LoomModel.i18nCatalog`. |
| ⑦ IR validate | Cross-check `text { … }` references resolve. |
| ⑧ codegen | Backends emit keys (not strings) for user-visible slots. React design packs gain an `IntlProvider` mount + `<FormattedMessage>` calls in body-walker output. Zod refines emit `message: <key>`. |
| ⑨ system | Write `out/.loom/messages.en.json`. Copy `<repo>/locales/*.json` into each emitted app's `src/locales/`. Merge in design-pack chrome catalogs under `chrome.<pack>.*`. |
| ⑩ write | Standard write phase; no change. |

The CLI subcommands live in `src/cli/i18n/` and share `enrichI18nCatalog`
with the codegen path — `ddd i18n extract` runs only phases ①–⑥ + the
catalog emit, skipping per-backend codegen.

## Phased implementation

**Phase 1 — extraction skeleton (≈1 week)**

- Mark user-visible slots in `walker-stdlib.ts` /
  `_walker/registry.ts`.
- Add `StringLitIR.userVisible`. Set in `lower-expr.ts`.
- New `enrichI18nCatalog` in `enrichments.ts`.
- Write `out/.loom/messages.en.json` in phase ⑨.
- Tests: catalog-extraction snapshot tests for `acme.ddd` and one
  scaffolded example.
- No backend behaviour change yet — keys are extracted but generated
  code still emits raw strings. Foundation only.

**Phase 2 — React runtime (≈1 week)**

- Add `react-intl` to React design packs' base manifest.
- React body-walker emits `<FormattedMessage id="..." defaultMessage="..." />`
  for user-visible string slots.
- Each emitted React app gets a generated `src/locales/`, an
  `IntlProvider` mount in its root, and a small `getLocale()` helper.
- Tests: byte-identical fixture diffs gated; one
  `LOOM_REACT_BUILD=1` smoke that the generated app type-checks with
  the new provider.

**Phase 3 — sync CLI + lock + three-way merge (≈1 week)**

- `ddd i18n init`, `sync`, `status`, `check`, `extract`, `prune`.
- `git merge-file --diff3`-style three-way merge for named keys;
  delete-old + add-new for content-hashed keys.
- Pre-commit hook recipe in docs.
- Tests: golden cases for each of the four merge scenarios; CI gate on
  a fixture repo with intentional `TODO:` markers.

**Phase 4 — chrome catalogs in design packs (≈1 week)**

- Mantine, shadcn, MUI, Chakra each gain `i18n/en.json` for their
  literal English strings. `.hbs` references the keys.
- Pack-catalog merge in phase ⑨ under `chrome.<pack>.*` namespace.
- One translated pack (Mantine `pt-BR`) as a worked example in tests.

**Phase 5 — Zod / invariant message keys (≈3 days)**

- Add `message:` slot to `InvariantIR` / `PreconditionIR`.
- `zod-refine.ts` emits `message: <stable key>`.
- Generated React error rendering: small `<DomainError>` component per
  React pack that resolves `error.message` through `react-intl`.
- `zod-i18n-map` import in generated app shell for built-in Zod errors.
- Tests: round-trip — DSL `precondition` → Zod refine emits key →
  catalog contains source → React renders translated text given a `fr`
  locale.

**Phase 6 — non-React backends (≈1 week per backend, parallelisable)**

- Hono: `src/i18n.ts` shim using `intl-messageformat`; `DomainError`
  serialisation includes `{ code, params }`.
- .NET: `IStringLocalizer` adapter reading JSON; error handler emits
  `code/params`.
- Phoenix: ICU → `.po` codegen step; Gettext at runtime; `:cldr_messages`
  optional.
- Per-backend obs-style smoke tests in `LOOM_TS_BUILD` / `LOOM_DOTNET_BUILD`
  / `LOOM_PHOENIX_BUILD`.

**Phase 7 — validator-message centralisation (≈2 days)**

- Move validator strings into `src/language/validators/messages.ts`
  keyed by existing diagnostic codes.
- No translation in v1, but the catalog is now overlay-ready.

Acceptance gate at every phase: byte-identical fixture output where
applicable, plus the existing `LOOM_*_BUILD` real-compilation suites
green.

## Open questions

1. **Locale folder location.** Project root (`locales/`) vs alongside
   each `.ddd` source file. Root is simpler for translators (one place
   to look); per-file scales better for multi-system repos. Default
   root; allow `--locale-dir` override.
2. **Should chrome catalogs ship with the pack version, or pinned per
   stack?** Argument for shipping with the pack: chrome strings are
   pack-specific. Argument for pinning: stack version controls the
   surface. Recommend with-pack; revisit if it churns.
3. **JSON5 / trailing commas for nicer diffs?** Strict JSON is simpler.
   The translator-friendly improvement is *one key per line, keys
   sorted*, which works in strict JSON.

## What this proposal does *not* settle

- **Localising compiler diagnostics.** Foundation laid by Phase 7;
  actual translation deferred.

The previously-open question on string-composition hygiene
(`"Order " + order.id` and friends) is closed by the companion note
[`i18n-strings.md`](./i18n-strings.md), which is required reading for
anyone implementing this proposal.
