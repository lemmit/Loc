# T4 — Eventing, workflow & temporal

*Weak-spot #4: scheduling was the language's temporal hole — no timers, no jobs, no saga deadlines. `timerSource` Phases 1 + 2 now close the timer half on all five backends (in-process scheduler + advisory-lock single-fire, plus durable Postgres-backed `cron:` drivers with store-coordinated single-fire, retry, and coalesce-once missed-run catch-up; only timezone/overlap sugar + saga deadlines still pending). Deployables remain runtime islands (in-process dispatch + outbox, but no cross-service transport). The saga/workflow core is otherwise solid and at full 5-backend parity.*

## M-T4.1 — `timerSource` (scheduling) — `partial` · **XL** · P1 (design-first)
The temporal hole. Time as an event source: a system-scope `timerSource` binding emitting tick events; workflows react via existing `on(e)`/`create(e) by` triggers — zero new workflow grammar (the `schedule every 5m {}` shape was explicitly rejected). Phase 1 (in-process scheduler + advisory-lock single-fire) AND Phase 2 (durable per-backend `cron:` drivers) are both landed on **all five backends**; only timezone/overlap sugar + saga deadlines remain.
**Phase 1 landed — all five backends:** grammar (`timerSource` SystemMember + `DURATION` terminal + system-wide `for:` scope), `TimerSourceIR`, cadence validators (`loom.timer-cadence` AST-level; `loom.timer-event-shape`/`-needs-state`/`-source-unbound` IR-level), print arm, obs catalog (`timer_fired`/`timer_skipped_overlap`/`timer_lock_contended`/`timer_emit_failed`). Owner derived from `migrationsOwner` (no stamp); byte-identical where unused. Per-backend driver, all sharing the tx-scoped `pg_try_advisory_xact_lock` single-fire (same FNV-1a→int32 key, so cross-backend replicas contend on one key) and dispatch through the existing in-process saga dispatcher:
- **Hono/node** (#1963) — `scheduler.ts`, node-cron (`cron:`) / setInterval (`every:`). `examples/timers.ddd`.
- **.NET** (#1986) — `TimerService : BackgroundService`, Cronos / `PeriodicTimer`, EF Core `SqlQuery<bool>`.
- **Java/Spring** (#1987) — `TimerScheduler @Component`, `@Scheduled(cron)` / `@Scheduled(fixedRate)`, `TransactionTemplate`. No new Gradle dep.
- **Python/FastAPI** (#1988) — `app/scheduling.py`, APScheduler `CronTrigger` / `IntervalTrigger`, `apscheduler>=3.10,<4`.
- **Elixir/Phoenix** (#1989) — one GenServer per timer, `Process.send_after` (`every:`) / `Crontab.Scheduler` (`cron:`, conditional `{:crontab}`); serialized GenServer loop makes overlap structurally impossible (no `timer_skipped_overlap`).
**Phase 2 landed — durable `cron:` drivers, all five backends.** A `cron:` timer becomes a durable, Postgres-backed job (store-coordinated single-fire across replicas + retry + coalesce-once missed-run catch-up); `every:` (sub-minute) timers stay the in-process advisory-lock path. Each backend uses its ecosystem's Postgres-native store; every driver was prototyped against Postgres, then runtime-verified on the *generated* stack (boot → cron fires → tick dispatches → saga persists; first-boot baseline + pushed-back-watermark catch-up asserted). Adds the `timer_catchup` obs event.
- **node/Hono** (#2009) — pg-boss (`boss.schedule`/`boss.work`, retryLimit 3) + a self-owned `loom_timer_runs` watermark for coalesce-once catch-up (pg-boss cron has no native backfill).
- **.NET** (#2101) — Hangfire + Hangfire.PostgreSql; standard cron, auto-schema, native overdue-on-start missed-run; `IRecurringJobManager`.
- **Java/Spring** (#2112) — JobRunr core (no SB4 starter → manual `JobRunrConfig`: SQL `StorageProvider` + background server + `scheduleRecurrently`).
- **Python/FastAPI** (#2132) — procrastinate `@app.periodic` tasks; the `periodic_defers` table is the store-coordinated single-fire + native coalesce-once catch-up ledger; worker runs in-process in the lifespan. Deps swap apscheduler → `procrastinate` + `psycopg[binary]`.
- **Elixir/Phoenix** (#2135) — Oban; a scheduler GenServer enqueues unique (per-boundary) Oban jobs (`unique` = the single-fire ledger, never re-runs a boundary) + a `loom_timer_runs` watermark for coalesce-once catch-up.
**Remaining:** timezone `in:` + `overlap: allow` sugar; Phase 3 (`missed: run`/`retry:` surface syntax, per-env cadence, saga-deadline sugar, `ddd verify` join).
**Runtime-verified (Hono, docker postgres):** the timer fires on cadence → tx-scoped `pg_try_advisory_xact_lock` (single-fire; no lock leak under concurrent load — the fix over a plain session lock, which leaks the unlock across pooled connections) → tick dispatched through the in-process channel dispatcher → `create(t) by t.<id>` reactor creates one saga instance per tick → persisted + readable via `/api/workflows/.../instances`.
**Note — reactor correlation:** a tick reaches a workflow only through a `channel` carrying it, and workflow event-consumers require an id-shaped correlation field; the scheduler mints tick fields (`newXId()` / `new Date()`), so `create(t) by t.<id>` spawns one instance per tick. Relaxing correlation for stateless recurring `on(t)` is a Phase-2/3 consideration.
**Note — delivery is at-least-once:** the advisory lock is a mutex around the body (no concurrent double-fire), but clock-skewed peers can fire sequentially, so reactors must be idempotent. Exactly-once-per-instant (durable claim / competing-consumer) is deferred to Phase 3 — not needed for the at-least-once contract.
