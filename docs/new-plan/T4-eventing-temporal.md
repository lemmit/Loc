# T4 — Eventing, workflow & temporal

*Weak-spot #4: scheduling was the language's temporal hole — no timers, no jobs, no saga deadlines. `timerSource` Phase 1 now closes the timer half on all five backends (in-process scheduler + advisory-lock single-fire; durable drivers + saga deadlines still pending). Deployables remain runtime islands (in-process dispatch + outbox, but no cross-service transport). The saga/workflow core is otherwise solid and at full 5-backend parity.*

## M-T4.1 — `timerSource` (scheduling) — `partial` · **XL** · P1 (design-first)
The temporal hole. Time as an event source: a system-scope `timerSource` binding emitting tick events; workflows react via existing `on(e)`/`create(e) by` triggers — zero new workflow grammar (the `schedule every 5m {}` shape was explicitly rejected). Phase 1 (in-process scheduler + advisory-lock single-fire) is done on **all five backends**; the durable-driver upgrade (pg-cron/poller, Quartz, Oban, Hangfire-analogue, APScheduler) + saga-deadline sugar is the remaining depth.
**Phase 1 landed — all five backends:** grammar (`timerSource` SystemMember + `DURATION` terminal + system-wide `for:` scope), `TimerSourceIR`, cadence validators (`loom.timer-cadence` AST-level; `loom.timer-event-shape`/`-needs-state`/`-source-unbound` IR-level), print arm, obs catalog (`timer_fired`/`timer_skipped_overlap`/`timer_lock_contended`/`timer_emit_failed`). Owner derived from `migrationsOwner` (no stamp); byte-identical where unused. Per-backend driver, all sharing the tx-scoped `pg_try_advisory_xact_lock` single-fire (same FNV-1a→int32 key, so cross-backend replicas contend on one key) and dispatch through the existing in-process saga dispatcher:
- **Hono/node** (#1963) — `scheduler.ts`, node-cron (`cron:`) / setInterval (`every:`). `examples/timers.ddd`.
- **.NET** (#1986) — `TimerService : BackgroundService`, Cronos / `PeriodicTimer`, EF Core `SqlQuery<bool>`.
- **Java/Spring** (#1987) — `TimerScheduler @Component`, `@Scheduled(cron)` / `@Scheduled(fixedRate)`, `TransactionTemplate`. No new Gradle dep.
- **Python/FastAPI** (#1988) — `app/scheduling.py`, APScheduler `CronTrigger` / `IntervalTrigger`, `apscheduler>=3.10,<4`.
- **Elixir/Phoenix** (#1989) — one GenServer per timer, `Process.send_after` (`every:`) / `Crontab.Scheduler` (`cron:`, conditional `{:crontab}`); serialized GenServer loop makes overlap structurally impossible (no `timer_skipped_overlap`).
**Remaining:** Phase 2 (timezone `in:` + `overlap: allow`; durable drivers per backend — Oban/Quartz/Hangfire/APScheduler-persistent replacing the in-process loop); Phase 3 (durability: `missed: run`, `retry:`, per-env cadence, `ddd verify` join).
**Runtime-verified (Hono, docker postgres):** the timer fires on cadence → tx-scoped `pg_try_advisory_xact_lock` (single-fire; no lock leak under concurrent load — the fix over a plain session lock, which leaks the unlock across pooled connections) → tick dispatched through the in-process channel dispatcher → `create(t) by t.<id>` reactor creates one saga instance per tick → persisted + readable via `/api/workflows/.../instances`.
**Note — reactor correlation:** a tick reaches a workflow only through a `channel` carrying it, and workflow event-consumers require an id-shaped correlation field; the scheduler mints tick fields (`newXId()` / `new Date()`), so `create(t) by t.<id>` spawns one instance per tick. Relaxing correlation for stateless recurring `on(t)` is a Phase-2/3 consideration.
**Note — delivery is at-least-once:** the advisory lock is a mutex around the body (no concurrent double-fire), but clock-skewed peers can fire sequentially, so reactors must be idempotent. Exactly-once-per-instant (durable claim / competing-consumer) is deferred to Phase 3 — not needed for the at-least-once contract.
Sources: [scheduling.md](../old/proposals/scheduling.md), weak-spots §4, completeness-audit Tier 1.

## M-T4.4 — Cross-deployable eventing (external brokers) — `in-flight` · **XL** · P2 (design signed off 2026-07-18)
`channelSource` parses but is "declared, not provisioned" — a microservice-shaped system is decorative. Provision brokers from `channelSource`, emit producer/consumer clients, `delivery: queue` competing-consumer semantics, service-to-service auth. Queue-semantics implementation sequences after M-T4.3 (producer rides the outbox relay). **Broker rollout pinned (2026-07-18): redis → rabbitmq → kafka, one slice each; NATS dropped** (stays a parseable storage type — `loom.channelsource-unsupported-transport` rejects it; ancestor-room routing rides Redis patterns / RabbitMQ topic wildcards, never Kafka). Licensing pinned free/OSS-only (design §6a: Valkey image, `apache/kafka`, no MassTransit). Excludes browser delivery (M-T1.10) but owns the `ChannelTransport` publish/subscribe seam both missions consume. Kickoff: [missions/M-T4.4-broker-eventing-brief.md](missions/M-T4.4-broker-eventing-brief.md); design (signed off): [missions/M-T4.4-broker-eventing-design.md](missions/M-T4.4-broker-eventing-design.md).
**Slice 1 landed:** deployable `channels:` wiring clause (grammar + `DeployableIR.channelSourceNames` + print arm + `channels` soft keyword), post-NATS compat matrix + `loom.channelsource-unsupported-transport` (AST), wiring validators `loom.channelsource-unbound`/`loom.deployable-channel-unrelated`/`loom.channel-consumer-unwired` (IR), shared vocabulary `src/util/channels.ts` (matrix + `loom.<ctx>.<chan>` addresses + consumer groups + CloudEvents envelope field pin), asyncapi `wiredBy:`. **Also unlocked cross-context reactors:** `on(e: X)`/projection-fold event refs now resolve system-wide (targeted scope arm mirroring `timerSource for:`) — previously context-local, which made the proposal's cross-deployable choreography inexpressible. Merged: #2035.
**Slice 2 built (PR #2050):** Redis `broadcast`/`ephemeral` transport — `valkey/valkey:8-alpine` compose sidecar (§6a licensing) + `LOOM_CHANNEL_*_URL` env injection, generated `http/channels.ts` on Hono (reference driver): `ChannelTransport` seam, CloudEvents envelope, ioredis pub/sub, producer tee enforcing the §4 delivery-uniformity rule (broker-bound events publish; co-located consumers receive via subscription), consumer loop into the in-process dispatcher, obs `channel_*` events. Cross-deployable vocabulary: a wired-but-foreign channel's events join the consumer's `domain/events.ts` (+ id brands); `create(e: X)` starters resolve foreign events (scope widening + system-wide member typing, the create-position twin of slice 1's `on(e: X)` arm); cross-context `X id` refs drop their FK in migrations (id-only linkage — the target lives in another schema/database). Landed as #2050 (e139804). Runtime-proven by the opt-in `channels-e2e` gate (`npm run test:channels`): two generated deployables, separate databases, docker valkey — an `Order.place()` emit on the producer spawns the correlated Fulfil instance + Shipment on the consumer.
**Slice 2b built (PR #2078):** the Python (FastAPI) leg of the same transport — generated `app/channels.py` (CloudEvents envelope codec over the DSL field names, redis.asyncio pub/sub, `publish_event`, consumer loop feeding `InProcessDispatcher`), `ChannelTeeDispatcher` inside `make_dispatcher` (a subscription-less pure producer gets a minimal `dispatch.py` so drained events still reach the tee), FastAPI lifespan boot/close wiring, wiring-gated `redis` dep, foreign event dataclasses + id brands + cross-context id imports in routes. Runtime-proven CROSS-BACKEND by the opt-in `channels-e2e-python` gate (`npm run test:channels-python`): the generated Hono producer publishes, the generated FastAPI consumer (own database) receives the same envelope id and spawns the correlated Fulfil instance + Shipment — two backends, one wire contract.
**Slice 3 in-flight:** RabbitMQ `queue` transport (Hono reference driver) — `rabbitmq:4-management-alpine` compose sidecar (MPL 2.0, §6a) + amqplib (MIT), §4 topology (per-address exchange, one durable queue per consuming deployable = `<address>.<deployable>`, replicas compete), manual ack → bounded retry → DLX `loom.dlx` → `loom.dlq.<channel>` parking + dead-letter obs event (co-designed with the M-T4.3 dead-letter item), envelope-`id` idempotency on consume; producer path split per design §5 — `queue`/`work` events ride the outbox relay (publish on drain), `queue`/`ephemeral` keeps the inline tee. Gate: `channels-e2e-rabbit` — exactly-one-of-N across 3 consumer replicas + DLQ parking on a poisoned message.
