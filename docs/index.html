orders.ddd 
aggregate Order {
  customerId: string
  status: OrderStatus
  placedAt: datetime
  contains lines: OrderLine[]

  derived total: Money =
    Money(lines.sum(l => l.subtotal.amount), "USD")

  invariant lines.count > 0 when status == Confirmed

  operation confirm() {
    precondition isMutable()
    precondition lines.count > 0
    status := Confirmed
    emit OrderConfirmed { order: id, at: now() }
  }
}

The vision

The model is the system.

Most DDD codebases are aspirational: a tactical pattern in the docs, a CRUD service in the repo. Loom closes that gap. The .ddd source is the single source of truth — every line of running code, in every deployable, is derived from it. Refactor the model, regenerate the system.

Domain-first, not framework-first

You write aggregates, invariants, operations, events, repositories — concepts from Domain-Driven Design, not from a web framework. The framework choice is a deployable attribute, not a design constraint.

Mechanical, not magical

Lowering goes through a fully-resolved IR, then per-platform Handlebars templates plus procedural builders. Generated output is idiomatic enough to read, debug, and grep — not framework lock-in.

One model, multiple platforms

The same aggregate compiles to ASP.NET + EF + Mediator and Hono + Drizzle + Zod. A built-in cross-platform OpenAPI parity check guarantees both backends present an identical wire shape.

The whole stack, including the UI

Per aggregate the generator emits list, detail, and create pages with React Query hooks, Zod parsing, Mantine forms, and Playwright page-objects keyed off stable data-testids. UI tests are written top-down, not selector-by-selector.

How it works

From .ddd source to a running stack.

The pipeline is fully type-safe end-to-end. Validation runs before any generator emits; a --dry-run mode prints the full write plan without touching disk.

01

Source

.ddd file authored against the Loom grammar.

02

AST

Langium parser & validator. Errors abort generation.

03

IR

Name resolution, type inference, semantic shaping — platform-neutral.

04

Templates

Per-platform Handlebars + procedural builders for TypeScript, .NET, React.

05

System

Multi-project tree with docker-compose.yml + healthchecks + e2e.

Three platforms, one model

Idiomatic output. Identical wire shapes.

Every backend produces the same aggregate root, repository, routes, and DTOs — but in the language and framework conventions you'd write by hand.

● TypeScript backend

Hono · Drizzle · Zod

HTTP via Hono with full type-validated routes, Drizzle ORM with master-detail load/save, and OpenAPI through @hono/zod-openapi.

  • Aggregate roots with private state & factories
  • Repositories with user-declared finds
  • Domain events dispatched on save
● .NET backend

ASP.NET Core · EF Core · Mediator

Controllers wired through martinothamar/Mediator, EF Core for persistence, Swashbuckle for OpenAPI — all scaffolded from your aggregates.

  • Entity-part classes with invariant checks
  • Derived properties & domain events
  • Cross-platform OpenAPI parity-checked
● React frontend

Vite · React Query · Mantine · Playwright

Per-aggregate list / detail / create pages, typed RQ hooks parsing responses with Zod, and Playwright page-objects keyed off stable test ids.

  • Master-detail when aggregates contain parts
  • Modal-form button per public operation
  • Generated UI e2e tests pass against the live stack
What you get

One generate. A full deployable stack.

From a single source, ddd generate system emits a runnable multi-deployable system with database init, healthchecks, OpenAPI specs, and end-to-end tests.

Aggregate roots, faithfully

Private state, factories, invariant checks, derived properties, and emitted domain events — in every backend.

Repositories with finds

find-by-id, save, find-all, plus user-declared finds — with full master-detail load and event dispatch.

HTTP routes & DTOs

Wire-shape DTOs identical across backends so the OpenAPI cross-check passes byte-for-byte.

List / detail / create UI

Mantine tables and modal forms, React Query hooks, Zod-validated request bodies — per aggregate, automatically.

Playwright page-objects

One class per page, keyed off stable data-testid attributes. Write top-down UI tests without crafting selectors.

Wired stack out of the box

docker-compose.yml with Postgres, every deployable, healthchecks, and a db-init SQL bootstrap — docker compose up -d and you're running.

CLI

Three commands. One model. Whatever shape you need.

Generate a single deployable for a focused service, or the full system for an integrated stack. Watch mode re-runs on save; --dry-run prints the plan with write / skip annotations.

~/acme
$ ddd generate system acme.ddd -o ./out
parse acme.ddd ............................. ok
validate ................................... ok (0 errors, 0 warnings)
emit api/ (.NET · Sales, Catalog) .. write 84

emit web_app/ (React · targets api) .... write 67
emit e2e/ (DSL e2e) ................ write 9
emit docker-compose.yml + db-init/ ......... write 2
cross-platform OpenAPI parity .............. match

$ docker compose -f out/docker-compose.yml up -d
