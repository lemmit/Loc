# Span-tracking emission — design + TS prototype (Milestone 15, phase 7 slice 2)

**Created:** 2026-07-06
**Status:** Prototype landed (TS/Hono only) — design pinned for the fan-out slices

See [`../proposals/source-map-and-debugging.md`](../proposals/source-map-and-debugging.md)
for the north star and the full phased roadmap; this doc is phase 7
("Char/expression granularity") slice 2. Slice 1 (`origin?` on every
`ExprIR` node, stamped at the `lowerExpr` chokepoint — #1743) already
landed; this slice is the emission side: turning those per-expression
origins into REAL generated-column mappings, prototyped end to end on the
TS/Hono backend.

## 1. Problem

Every construct- and statement-granular region recorded so far
(`SourceMapRecorder.file` / `.fragment`) reports a LINE range. `ExprIR` now
carries its own `origin?` (phase 7 slice 1), but nothing yet turns that
into a generated COLUMN — the Source Map v3 sidecar (`sourcemap-v3.ts`)
still emits every mapped line at a hardcoded column 0, so a debugger or
`ddd trace` can point at "the line" but never "the fifteen characters on
that line that came from `customerName`".

## 2. Pinned architecture — level-wise anchoring in `renderExprWith`

The insight: `src/generator/_expr/target.ts`'s `renderExprWith` already
renders every child via its local `r` closure BEFORE handing the child
**strings** to the leaf method (`t.member(recv, e, ctx)`,
`t.binary(left, right, e)`, …). At that point the dispatcher holds, in the
same stack frame, both (a) each child's already-rendered text and (b) the
leaf's freshly-composed output built from those texts. That is exactly the
information needed to place a child's marks inside the parent's text — no
new bookkeeping, no rewritten leaves.

`renderExprWithMarks` (`src/generator/_expr/target.ts`) mirrors the
17-arm switch/recursion byte-for-byte, but returns `{ text, marks }`
instead of a bare string:

- Recurse into every child via the same function, obtaining its `{ text,
  marks }`.
- Call the **same leaf method** with the children's `text` fields — leaves
  never change; a target table needs zero modification to support this.
- **Anchor** each child's marks into the parent's composed text: find the
  child's `text` inside the parent's output via `indexOf`. Exactly one
  occurrence ⇒ shift the child's marks by the match offset. Zero or more
  than one occurrence ⇒ **skip that child's marks** — an ambiguous anchor
  is a guess, not a fact, so we drop it rather than fabricate a column.
  This is the existing `SourceMapRecorder.fragment()` exact-text-anchor
  discipline, applied one level finer (per sub-expression instead of per
  statement).
- Append the node's own mark, `{ start: 0, end: text.length, origin:
  e.origin }`, when `e.origin` is set.

Because anchoring happens **one level at a time**, same-text SIBLINGS
skip (`a + a` — both operands render `"a"`, so `"a"` occurs twice inside
`"a + a"` and both anchors are dropped) while a DEEPER repeat several
levels down still resolves cleanly: `count > 0 && count < max` composes
`"count > 0"` and `"count < max"` as its two immediate children (each
already fully anchored internally — `"count"` occurs once inside `"count
> 0"`, once inside `"count < max"`), and those two composed strings are
distinct, so the outer `&&` anchors both without ambiguity. The composer
never has to reason about the WHOLE expression's text at once — each
level only ever searches within its own immediate parent's text.

`renderExprWith` itself is untouched (still the byte-cheap default path);
`renderExprWithMarks` is a parallel entry a caller opts into only when a
`SourceMapRecorder` is actually threaded in. No allocation happens on the
flag-off path.

### Alternatives considered and rejected

- **Offset-composing leaf rewrite.** Change every `ExprTarget` leaf method
  to return `{ text, offsets }` instead of a bare string, so leaves
  compute their own child-offset bookkeeping. Rejected: touches all 5
  backends × 17 arms = 85 leaf methods, each now responsible for tracking
  where it placed each argument inside its own string formatting — a
  large, invasive, and easy-to-get-subtly-wrong change to code that has
  been stable and byte-identical-gated for multiple milestones. The
  level-wise composer gets the same information for free from the
  dispatcher's existing recursion, so leaves never need to know about
  marks at all.
- **Marker injection** (embed sentinel tokens like `\0<id>\0`
  around sub-expressions during rendering, then strip and record their
  positions in a post-pass). Rejected: no precedent anywhere in this
  codebase's emission pipeline (which is plain string concatenation via
  `lines()`/template literals, never token-annotated), and a missed strip
  — an escaped string, a regex literal, a template-literal interpolation
  — silently corrupts generated bytes that are otherwise gated
  byte-identical. The risk of shipping corrupted source is worse than the
  cost of the honest "skip when ambiguous" behavior above.
- **Whole-chunk post-hoc anchoring** (keep emitting today's line-level
  fragment, then separately re-scan the fragment's *rendered text* for the
  RHS token via a single outer `indexOf`, without recursing into
  sub-expressions at all). Rejected: fails on the common case of a SHORT
  expression whose rendered text recurs elsewhere in the same fragment
  (e.g. a one-character `let x = a` inside a body that also reads `a`
  elsewhere) — no way to disambiguate without recursing level-by-level the
  way the composer above already does. The chosen design *is* this
  approach, just applied recursively instead of once.

## 3. Wire + region model — additive, no version bump

- `SourceMapRegion` (`src/generator/_trace/sourcemap.ts`) gains an
  OPTIONAL `targetCol?: [number, number]` — **1-based, half-open**
  `[startCol, endCol)`: `startCol` is the column of the first marked
  character, `endCol` is one past the last. This mirrors the 1-based
  convention `offsetToLineCol` already returns everywhere else in this
  file (`target`'s line numbers are also 1-based); the Source Map v3
  boundary is the one place it gets converted to 0-based, exactly the
  same treatment `target`'s lines already receive at that boundary.
  Meaningful only when `target[0] === target[1]` (a single-line region).
- `.loom/sourcemap.json` **stays version 1** — `targetCol` is additive on
  `WireRegion` (`src/system/sourcemap.ts`); a consumer reading only
  `target` is unaffected.
- `renderSourceMapV3` (`src/system/sourcemap-v3.ts`): a generated line
  covered by one or more regions carrying `targetCol` now emits ONE
  SEGMENT PER SUCH REGION, sorted by column, each pointing at its own
  precise (generated col) → (source line, col) pair — `genCol` resets to
  0 per line group per the v3 spec, `sourceIndex`/`sourceLine`/`sourceCol`
  keep running deltas across the whole `mappings` string as before. A line
  with no `targetCol`-bearing region keeps today's construct-granular
  single segment at column 0 (`narrowestRegion`'s existing tie-break),
  untouched.

## 4. Prototype scope — TS/Hono only, `let`/`assign`/`return` RHS only (SHIPPED — widened in slice 4)

Full statement-shape coverage was out of scope THIS (slice 2) prototype
(mirrors the .NET `#line` weave's own narrowing from phase 6a — see
`narrowedOrigin`/M7); **M17 phase 7 slice 4 widened it** to every
expression-bearing `StmtIR` kind — `precondition`/`requires` predicates,
`add`/`remove` values, bare `expression` statements, `call` args, and
`emit` field values, alongside the original `let`/`assign`/`return` — still
TS/Hono only (see §6). The marks-carrying path (as it stood at slice 2;
still accurate, just fanned across more `StmtIR` kinds now):

- `src/generator/typescript/render-expr.ts` exports
  `renderTsExprWithMarks` — a thin sibling of `renderTsExpr` that runs the
  SAME `TS_TARGET` leaf table through `renderExprWithMarks` instead of
  `renderExprWith`.
- `src/generator/typescript/render-stmt.ts` exports
  `statementExprMarks(stmt, chunk)` — for `let`'s initializer, `assign`'s
  value, and `return`'s value ONLY (every other statement kind returns no
  marks this slice), it renders the RHS a second time via
  `renderTsExprWithMarks`, then locates that text inside the
  ALREADY-rendered chunk via the identical one-occurrence anchor rule
  `fragment()` uses. Absent or ambiguous ⇒ honest skip (e.g. a
  provenance-instrumented chunk that repeats the RHS text inside its
  `__prov_N` leaf-snapshot array skips here for exactly that reason — the
  chunk-level anchor is just as honest as the expression-level one).
- `src/generator/_trace/sourcemap.ts`'s `statementSubRegions` gains an
  optional 4th parameter, `exprMarks: readonly (readonly ChunkMark[])[]`
  (one entry per statement, keyed the same way `chunks` already is) —
  when present, each mark's chunk-relative character offset converts to a
  (line, col) via the existing shared `offsetToLineCol`, and becomes an
  ADDITIONAL fine-grained `SourceMapSubRegion` (carrying `col`) layered
  alongside the existing coarse per-statement region. `fragment()` copies
  `sub.col` straight into the pushed region's `targetCol`.
- `src/generator/typescript/emit/aggregate.ts`'s per-operation loop
  computes `exprMarks` ONLY inside the existing `if (opFragments && …)`
  recording branch — a flag-off run never calls
  `statementExprMarks`/`renderTsExprWithMarks`, so it pays zero extra
  work or allocation, exactly like every other sourcemap bracket in this
  codebase.

This is proved end to end on the SOURCE fixture's `Order.confirm()`
body — `let note = customerName` renders as
`const note = this._customerName;`, and the `customerName` ref's own
mark (a leaf node with no children, so its only mark is its own whole-text
one) now anchors `this._customerName` at its REAL generated column, back
to the exact `customerName` token in the `.ddd` source.

## 5. Formatter-ordering question — RESOLVED

Proposal §10 left this open: "does map-build run after Biome on generated
TS, or do we re-anchor via markers post-format?" It's moot: **there is no
post-emission reformatting step in this pipeline.** `test:biome-gen`
(`test/e2e/generated-biome.test.ts`, gated by `LOOM_BIOME=1`) runs
`biome lint` — a CHECK, never `biome format` / `--write` — against
generated output; nothing in `generate system`'s write path (`src/cli/
main.ts` phase ⑩) runs a formatter over the emitted bytes either. The
bytes the recorder measures ARE the bytes written to disk; there is no
later pass that could move a column out from under a recorded mark. This
closes the open question in the proposal (§10 updated to point here)
without requiring any re-anchoring machinery.

## 6. Fan-out plan (later slices, not built here)

- **Slice 3 — `ddd trace` columns (SHIPPED, M16 phase 7 slice 3):**
  `src/trace/` consumes the `targetCol` regions this slice emits. A
  V8/Node frame's column (the only native crash-log dialect that carries
  one) selects the NARROWEST `targetCol` region containing it
  (`targetCol[0] <= col < targetCol[1]`) on the frame's line, and the
  annotation prints the exact `.ddd` `path:line:col` of that
  sub-expression instead of just the statement's line. Column-less logs
  (and columns matching no `targetCol` region) resolve byte-identically
  to before: `targetCol`-bearing regions are EXCLUDED from the
  line-narrowest fallback, so a fine expression region never wins by
  line-width accident. See `src/trace/{frames,resolve,annotate}.ts`.
- **Other four backends** (.NET, Phoenix/Elixir, Python, Java): each
  backend's own `render-expr.ts` already implements `ExprTarget`, so each
  gets a `renderExprWithMarks(e, ITS_TARGET, ctx)` call for free from the
  shared dispatcher — no new per-backend anchoring logic, only (a) a
  thin `render<Lang>ExprWithMarks` wrapper per backend and (b) that
  backend's own statement renderer growing the same
  `statementExprMarks`-shaped helper. .NET is the natural second target —
  it already narrows `let`/`assign`/`return` for its `#line` directive
  weave (phase 6a), so the RHS-location logic is already half-written
  there.
- **DAP consumption** (phase 8): once every backend reports real columns,
  a debug adapter can set a breakpoint on a `.ddd` line/column and resolve
  it to the exact generated statement AND sub-expression, rather than just
  "somewhere on this generated line" — the missing piece phase 8 needs
  that this slice supplies for TS/Hono.
- **Statement kinds beyond `let`/`assign`/`return`** (SHIPPED, M17 phase 7
  slice 4): `markableExprsOf` in `src/generator/typescript/render-stmt.ts`
  now covers `precondition`/`requires` predicates, `add`/`remove` values,
  bare `expression` statements, `call` args, and `emit` field values —
  same anchoring shape, each expr anchored independently of its siblings.
  The biggest payoff is `precondition`/`requires`: those are the actual
  THROW sites a Node stack frame points at on a failed invariant, so `ddd
  trace` (slice 3) now resolves a thrown precondition to the exact
  predicate text, not just the statement's line.
- **Other four backends remain deliberately deferred** (scout-confirmed,
  M17 phase 7 slice 4): .NET, Phoenix/Elixir, Python, and Java have no
  live consumer of a generated column today (no `#line`-style per-column
  weave reads one, and `ddd trace`'s column-aware resolution is
  Node/V8-frame-specific) — fanning `statementExprMarks` out to them has
  no payoff yet, so it stays scoped to TS/Hono until a consumer exists.
