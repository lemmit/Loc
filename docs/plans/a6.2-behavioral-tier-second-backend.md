> **Bug found on day one — and fixed.** An operation with a cross-aggregate
> `X id` param (`Order.addLine(productId: Product id)`) emitted

> imported only the aggregate's own id types

> `mypy --strict` name-defined error, had a fixture exercised it). Fix: the
> route import collector (`routes-builder.ts`) now draws id candidates from

> reached via an operation param or a contained-entity field is imported. The

## The goal, precisely

The runtime-semantics contract (`conformance-semantics.md`) has nine RS-rules.
Three are **T0/static** and already gate per-PR (RS-2 enum casing, RS-3 wire
no-leak, RS-5 absence-match). The other six are **behavioral** — they can only
be observed by booting a backend and round-tripping HTTP:

- RS-1 (camelCase wire keys, both directions), RS-4 (declared temporal
  round-trip), RS-6 (bool-default materialization), RS-7 (jsonb-VO subfield
  survival), RS-8 (association persistence round-trip), RS-9 (400/422 split).

Today those are proven **only for Hono/node**, via the docker-free in-process
behavioral tier (`test/behavioral/run.mjs`: PGlite + `app.fetch`). Every other
backend's runtime behavior waits for the nightly/label docker `conformance-full`
leg — which is exactly why the #1620–#1660 wire-parity avalanche landed on
`main` and was chased after the fact. **A6.2 = get one non-Hono backend
round-tripping these rules per-PR.**

## The constraint that makes this non-trivial

The node tier is docker-free *only* because of a coincidence that does not
generalize: PGlite is Postgres-in-WASM running in the same Node process, and
Hono dispatches via `app.fetch(req)` with no socket. No other backend has an
in-process Postgres:

- Python/SQLAlchemy, .NET/EF, Java/JPA, Elixir/Ecto all speak to a **real
  Postgres over a socket**, and the generated DDL is Postgres-specific (jsonb,
  `pgEnum`, `gen_random_uuid()`), so SQLite-substitution is not viable.

So a second backend needs **(a)** its language runtime and **(b)** a real
Postgres process. The question is how cheaply, and whether it still counts as a
per-PR gate.

## Two reuse points that make it tractable

1. **Dispatch is already abstracted.** `run.mjs` builds
   `dispatch = async (req) => app.fetch(...)` and hands it to
   `loadApiTests({ source, compile, dispatch })` (`web/src/testing/run-api-tests.ts`).
   The emitted `e2e/<Sys>.e2e.test.ts` is written against the **HTTP contract**,
   not against Node — it is backend-agnostic. Swapping `app.fetch` for a real
   `fetch("http://127.0.0.1:PORT" + path)` runs the *same assertions* against
   any booted backend. **No second per-backend test suite is needed for the api
   tier.**

2. **The Python boot recipe already exists.** `test/e2e/observability-events-python.test.ts`
   already: provisions Python 3.12 via `uv` (self-contained, no system Python),
   `spawn`s uvicorn against the generated project, waits for the port to listen
   (`net.connect`), and takes Postgres from **docker by default OR an external
   instance via `LOOM_OBS_PG_URL`** (the docker-less hook). That boot+wait+teardown
   is directly liftable.

## Options considered

| Option | Postgres | Cost per PR | Gates RS-rules? | Verdict |
|---|---|---|---|---|
| **A. HTTP-dispatch the TS e2e against booted Python** (recommended) | GH Actions `services: postgres` sidecar | uv install + uvicorn boot (~1–2 min warm) | **Yes** — RS-1/4/7/8/9 on Python | **Recommended** |
| B. Same, but embedded-PG binary (no service container) | `pytest-postgresql` / embedded binary | binary download + init | Yes | Fallback if service containers are unwanted; more moving parts |
| C. Promote a `src/ir/**` subset of `conformance-parity` to required | (its own) | already runs | **No — structural only** | Insufficient: the OpenAPI diff is casing-tolerant and value-blind (the whole reason the RS contract exists) |
| D. Second in-process backend (SQLite-shim Python) | in-process | cheapest | No — DDL is PG-specific, would diverge | Not viable |

Option C is the audit's stated fallback; it is worth knowing it does **not**
close the runtime-value gap — it re-gates the *structural* contract that already
has a gate. It's a consolation, not the goal.

## Recommended approach (Option A)

**Boot the generated Python FastAPI backend against a Postgres service and run
the existing emitted api-tier e2e over real HTTP.**

Why Python as the first second backend: cheapest runtime to provision (`uv`
self-installs the interpreter — no SDK, JVM+Gradle, or BEAM), and the boot
harness is already written (obs-python).

### Implementation sketch

1. **Runner (`test/behavioral/`):** factor `run.mjs`'s dispatch so the api tier
   can target either `app.fetch` (node, in-process) or an HTTP base URL. Add
   `run-python.mjs` (or a `--backend=python` mode) that:
   - generates the corpus system, locates the `platform: python` deployable
     (add one to the corpus systems, or a parallel `corpus-python.json`),
   - runs the emitted migrations against the service Postgres (`uv run` the
     project's Alembic/DDL step — confirm the generated migrate entrypoint),
   - `spawn`s uvicorn, waits for the port (lift from obs-python),
   - dispatches the emitted `e2e/<Sys>.e2e.test.ts` assertions via HTTP,
   - tears down.
   Scope v1 to the **api tier only**. The unit tier is emitted in TS and runs
   in-process against the domain layer — the Python analogue is the generated
   pytest domain suite, a separate (larger) lift; defer it.

2. **CI (`.github/workflows/behavioral-e2e-python.yml`,** or a leg on the
   existing `behavioral-e2e.yml`): a `services: postgres` sidecar +
   `LOOM_OBS_PG_URL` pointed at it, `uv` setup, `node test/behavioral/run-python.mjs`.
   Path-filter on `src/ir/**`, `src/generator/python/**`, `src/generator/_expr/**`,
   `src/platform/python.ts` — the surfaces whose drift it catches.

3. **Corpus:** the `node-only` constraint in `corpus.json` exists for the
   in-process dispatch; the HTTP path removes it for the api tier. Either add a
   `platform: python` deployable to the existing single-backend systems or carry
   a `backends: ["node","python"]` field per case.

### What it newly gates

RS-1 (camelCase both directions — the #1620 class), RS-4 (temporal round-trip),
RS-7 (jsonb-VO), RS-8 (associations), and RS-9 (400/422) — **on Python, per-PR**.
RS-6 (bool-default) too if a corpus system exercises it. This is precisely the
drift class the node tier can't see and conformance-full catches a day late.

## Open questions — resolved in v1

1. **Migrate entrypoint — resolved.** No separate step needed: the generated
   app runs `run_migrations()` in its FastAPI `lifespan` on startup (and
   `python -m app.db.migrate` exists out-of-band). Booting uvicorn migrates
   itself; the runner only sets `DATABASE_URL` + boots.
2. **`services: postgres` as a per-PR cost — resolved (approved).** v1 uses a
   GitHub Actions postgres service sidecar (the lightweight service container,
   not the heavy compose stack). Option B (embedded binary) stays the fallback
   if that ever proves too costly.
3. **Corpus shape — resolved.** A parallel `test/behavioral/corpus-python.json`
   + a dedicated `corpus-python/sales.ddd` fixture (a python-backed cut of the
   Sales domain), keeping the node corpus untouched and dispatch unambiguous.
