# Coordinated fixture re-baseline — operational guide

> Operational spec. Status: the capture script
> (`scripts/capture-baseline-fixture.mjs`) and byte-for-byte fixtures
> under `test/fixtures/` exist today; this doc pins *when* and *how* to
> re-baseline so the big grammar moments (M1, M2, M3, Lifecycle-1,
> Inheritance) don't each reinvent the dance.

## What a fixture is

`test/fixtures/` holds **byte-for-byte snapshots of generated output**
— excluded from vitest discovery (`vitest.config.ts`), used as
regression anchors. A normal PR must leave them byte-identical: if a
change alters generated bytes without an explicit re-baseline, the
fixture diff fails CI, which is exactly the signal we want for
*accidental* generator drift.

Some changes alter generated bytes **on purpose** (a new wire field, a
renamed route, a different DTO shape). Those are the coordinated
re-baseline moments. The rule:

> **A re-baseline is a deliberate, isolated, single-purpose PR.** Never
> fold a fixture re-capture into a feature PR — the reviewer can't tell
> intended drift from a bug when they're mixed.

## The re-baseline moments

| Tag | Trigger | Expected drift |
|---|---|---|
| **M1** | payload P3 + P4 (carrier generics + tagged unions) | new wire payload shapes; union `kind` fields |
| **M2** | exception-less A1 + A2 + A3 | `option`/`error` fields; ProblemDetails routes |
| **M3** | exception-less A4 (`Repo.getById` → `T or NotFound`) | every repository + route touching `getById` |
| **Lifecycle-1** | lifecycle grammar + `OperationIR.kind` | route slugs, form contracts (create/operation/destroy) |
| **Inheritance** | abstract aggregates + `inheritanceUsing(…)` | repository SQL / EF / Ash table layout (TPH/TPC) |

M1/M2 bundle multiple sub-phases precisely so there is **one** drift
event, not one per sub-phase (see global-plan "Coordinated single-PR
moments"). M3 and Inheritance each touch a wide blast radius and get
their own isolated re-baseline.

## The procedure

1. **Land the feature behind the drift first**, with its fixtures
   *expected to fail*. CI red on fixture diff only (all other suites
   green) is the gate that the drift is isolated to fixtures.
2. **Open the re-baseline PR** tagged `rebaseline-<tag>:` (the PR-type
   taxonomy from global-plan §0.6). Title states the drift in one line.
3. **Re-capture**:
   ```bash
   node scripts/capture-baseline-fixture.mjs        # all fixtures
   # or scope to the affected case(s) where the script supports it
   ```
4. **Review the diff as the deliverable.** The PR *is* the generated-byte
   delta. Reviewer reads it as "is every changed byte explained by the
   feature?" — unexplained drift = a bug in the feature PR, kicked back.
5. **Cross-backend check.** Run the relevant `LOOM_*_BUILD=1` gates and
   the OpenAPI/wire-spec parity (`conformance-parity.yml`) — a wire
   change must land on **all** backends in the same drift, or the parity
   gate fails. The wire-spec diff (`.loom/wire-spec.json`) is the
   contract-change record; review it explicitly.

## What never triggers a re-baseline

- Pure refactors / seam extractions — these are guarded by the
  byte-identical gate *as a feature* (the WalkerTarget extraction shipped
  under exactly this rule: every per-seam slice proved byte-identical
  output). If a refactor changes bytes, it's not a refactor.
- IR-internal changes with no emit consumer.

## Why isolate

A mixed feature+rebaseline PR hides bugs: a stray byte change buried in
a 4,000-line fixture diff reads as "expected churn". Isolating the
re-capture turns the diff into a reviewable artefact and keeps `git
blame` on generated fixtures meaningful — every fixture line traces to
exactly one `rebaseline-<tag>:` commit.
